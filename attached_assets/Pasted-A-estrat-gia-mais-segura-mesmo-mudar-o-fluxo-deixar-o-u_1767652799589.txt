A estratégia mais segura é mesmo mudar o fluxo: deixar o usuário se cadastrar diretamente na tela inicial (com login e senha + confirmação), fazer o primeiro login e, logo depois, abrir automaticamente a tela de “Novo Usuário” para completar os dados administrativos, sem mexer em nada que já funciona no restante do sistema.[1][2]

## Ideia do novo fluxo

- Manter todo o código atual de gestão de usuários na rota/menu existente, apenas **reutilizando** a lógica de “Novo Usuário” após o primeiro login do cadastro simplificado.[1]
- Remover (ou ignorar) a etapa de envio automático de e‑mail com login e senha provisória, já que essa parte não está funcionando de forma confiável.[3]

## Comportamento desejado na tela inicial

- A tela da imagem passa a ter duas áreas: “Já tenho conta” (login normal) e “Criar conta”.[4]
- Em “Criar conta”, o usuário informa: nome, e‑mail, senha e confirmação de senha; ao salvar, o sistema cria o registro mínimo de autenticação e já faz login automático ou redireciona para login com esses dados.[3]

## Passos após o primeiro login

- Depois do primeiro login de quem acabou de se cadastrar, o sistema redireciona automaticamente para a mesma tela de “Novo Usuário” usada hoje em Usuários > Gestão de Usuários > + Adicionar Usuário.[5]
- Essa tela é carregada já com o e‑mail e o nome trazidos do cadastro de login, o usuário completa os campos obrigatórios, salva, e só então o sistema libera o restante dos menus para uso.[6]

## Prompt sugerido para o Replit Agent

Use este texto (ajuste só nomes de arquivos/rotas se necessário):

> “Você é o desenvolvedor responsável por **implementar novas funcionalidades sem alterar nada do que já funciona** no meu sistema atual. A aplicação já possui:  
> - Tela de login funcional, que autentica usuário por e‑mail e senha.  
> - Módulo de gestão de usuários acessível por `Sidebar > Usuário > Gestão de Usuários > + Adicionar Usuário`, com a tela de `Novo Usuário` já funcionando.  
> - Toda a parte de permissões, menus e acesso ao sistema já está correta e **não pode ser modificada**.  
>  
> Preciso que você implemente **apenas** o seguinte fluxo, preservando o código existente e criando novos componentes/rotas quando necessário:  
>  
> 1. Transformar a tela inicial (a tela atual de login, arquivo X) em uma tela com duas seções:  
>    - ‘Já tenho conta’: mantém exatamente o comportamento atual de login.  
>    - ‘Criar conta’: novo formulário de cadastro simplificado.  
> 2. O formulário ‘Criar conta’ deve ter os campos: nome completo, e‑mail, senha, confirmação de senha.  
>    - Validar que senha e confirmação são iguais.  
>    - Validar formato básico do e‑mail.  
>    - Ao enviar o formulário, criar um novo registro de usuário **apenas na tabela/coleção de autenticação/login**, sem mexer na tabela de usuários administrativos já usada pelo módulo de gestão de usuários.  
>    - Não enviar e‑mail automático nem gerar senha provisória.  
> 3. Após o cadastro bem-sucedido, o usuário deve:  
>    - ou ser autenticado automaticamente e redirecionado,  
>    - ou ser redirecionado para a tela de login com o campo e‑mail já preenchido.  
> 4. No primeiro login desse novo usuário, o sistema deve verificar se já existe um registro correspondente na tabela usada pelo módulo `Usuário > Gestão de Usuários`.  
>    - Se **não existir**, redirecionar automaticamente para a tela já existente de `Novo Usuário`, reutilizando o mesmo componente/rota atual.  
>    - Pré-preencher o e‑mail e o nome com os dados do cadastro de login.  
>    - Ao salvar o `Novo Usuário`, liberar automaticamente o acesso normal ao sistema (menus, dashboards etc.), sem alterar as regras de permissão já implementadas.  
> 5. Não remova nem modifique nenhuma funcionalidade existente, especialmente:  
>    - Login atual de usuários já cadastrados.  
>    - Tela e lógica de `Gestão de Usuários` e `Novo Usuário`.  
>    - Permissões, papéis e demais regras de negócio.  
> 6. Caso seja necessário criar novos arquivos, rotas ou serviços, documente no código onde foi feita cada alteração e mantenha a arquitetura existente.  
>  
> Antes de qualquer mudança crítica, analise o projeto para garantir que o novo fluxo convive com o fluxo antigo, sem quebrar nada.”  

Esse prompt deixa claro o problema, a nova estratégia e a restrição de não alterar o que já está funcionando, o que tende a gerar um resultado mais previsível no Replit Agent.[2][7]
